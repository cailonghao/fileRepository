# 权限系统与安全体系深度分析报告

## 1. 设计哲学 (Design Philosophy)
Kapok-Main3 采用了一种 **“侵入性极低、高度集成化”** 的权限设计逻辑。它摒弃了 Spring Security 或 Shiro 等大而全的传统框架，转而通过 **自定义注解 (`@AppCode`) + AOP 切面 (`AppCodeAspect`)** 实现了对身份认证、权限授权和审计日志的一站式管理。

## 2. 数据层设计 (RBAC Model)
系统基于标准角色访问控制（RBAC）模型，并通过冗余表（或视图）提升了查询效率。

### 2.1 核心表结构
- **`sys_user`**: 系统用户主体。
- **`sys_role`**: 角色定义，作为用户与权限之间的中间媒介。
- **`sys_menu`**: 权限资源池，通过 `type` 区分：
    - `1 & 2`: 目录与菜单（UI 层面）。
    - `3`: 按钮/接口权限（策略层面，对应代码中的 `perm` 标识符）。
- **`sys_user_menu`**: **核心优化表**。该表平铺了用户的所有有效权限，直接支持通过 `userId + perm` 进行 O(1) 或 O(log n) 级别的快速检索。

## 3. 实现原理 (Implementation Details)

### 3.1 注解驱动：`@AppCode`
在 Controller 层，每个受保护的方法都使用此注解描述其安全属性：
```java
@AppCode(perm = "main:CrmQuote:save", log = "报价单-保存", client = "PC端")
```
- **`perm`**: 定义所需的权限字符串。
- **`log`**: 激活操作审计，定义日志标题。
- **`client`**: 声明端类型（PC / APP），实现业务隔离。

### 3.2 拦截核心：`AppCodeAspect`
`around` 通知是整个安全体系的交通枢纽，其处理流程如下：
1. **JWT 校验**: 提取 Header 中的 Token，验证其签名合规性及有效期。
2. **端路由**: 根据注解中的 `client` 值，自动路由至 `SysUserService`（PC 权限池）或 `AppUserService`（移动权限池）。
3. **权限比对**: 查询 `sys_user_menu` 确认当前用户是否被授予了对应的 `perm`。
4. **环绕审计**: 无论权限校验通过与否，都会自动收集请求参数、耗时及响应结果。

## 4. 安全特性与亮点

### 4.1 双端隔离授权
系统支持同一套用户体系在 PC 和 APP 上表现出不同的权限。例如，某个财务接口可能只允许在 PC 端调用，而考勤打卡只允许在 APP 端调用，这一机制通过 AOP 层面的 `client` 过滤实现。

### 4.2 一体化审计追踪
权限系统与日志系统是共生的。每一次成功的越权尝试被拦截，或每一次合法的业务改动，都会在 `sys_user_log` 中留下包含 IP、地址（通过 IP 库识别地域）、请求报文在内的完整历史记录。

### 4.3 粒度可定制性
- **功能权限**: 控制是否能点击按钮。
- **数据权限 (Implicit)**: 虽然 `@AppCode` 负责功能权限，但其内部往往配合 `RequestUtil.getUserId()` 引入了门店级或部门级的数据隔离策略。

## 5. 总结与评估
Kapok-Main3 的权限管理系统非常契合快速开发的业务场景。其设计既保证了中大型 ERP/CRM 所需的严谨安全性，又通过无状态 JWT 和 AOP 拦截确保了高并发下的性能表现，是一套高性能且易于维护的企业级安全方案。
