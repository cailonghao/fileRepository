# 报价单与后续流转 (Quote to Sale Flow) 分析报告

## 1. 业务全链路概览
本系统实现了从前期报价到正式订单，再到后期采购的完整自动化流转。
**核心链路**: `报价单 (CrmQuote)` -> `销售单 (CrmSale)` -> `采购单 (CrmPurchase)`。

## 2. 报价单生命周期 (Quote Lifecycle)

### 2.1 创建与校验 (`save`)
- **必填项**: 客户信息、负责人、合同模板、有效产品明细。
- **金额计算**: `总金额 = 产品合计 - 优惠金额 + 运费`。

### 2.2 核心审核关键点 (`check`)
- **价格权限校验**: 这是系统最核心的逻辑之一。
    - 系统会提取报价单中所有产品的价格。
    - 与当前登录人的 **“最高价格权限”** 进行比对：
        - 权限分为：部门管理员 (`isDeptOwner`)、门店店长 (`isShopOwner`)、分销商老板 (`isDistOwner`) 以及系统超级管理员 (`Boss`)。
    - 如果单据中任一产品的折扣低于用户的授权底线，审核将被拦截并报错“价格权限不足”。
- **状态流转**: 审核通过后状态从 `1 (待审)` 变为 `2 (已审)`。

### 2.3 客户确认 (`confirm`)
- 已审核的报价单可交由客户确认。
- 支持上传确认凭证图（如签字照片），并触发合同图片的重新生成 (`infoImage`)。

## 3. 生成销售单逻辑 (`createSale`)

### 3.1 触发条件
- 报价单必须处于 `已审核 (status=2)` 状态。
- 系统会通过 `relt_bill_id` 自动校验是否已存在关联销售单，防止重复生成。

### 3.2 数据映射 (Data Mapping)
- **主体拷贝**: 将报价单的客户信息、金额、部门/门店归属关系完整克隆到销售单。
- **明细提取**: 遍历 `CrmQuoteProduct` 并在 `CrmSaleProduct` 中创建对应的行项目。
- **状态初始化**: 新生成的销售单默认处于 `待审 (status=1)` 状态。

## 4. 后续流转：从销售到采购库运
- **生成采购单**: 在 `CrmSaleFacade` 中存在类似的 `createPurchase` 逻辑，允许根据销售订单一键生成采购需求。
- **库存闭环**: 销售单关联 `CrmSaleOutStore` (销售出库)，采购单关联 `CrmPurchaseInStore` (采购入库)，实现了实物库存与价值流的匹配。

---
**总结**: 系统的流转逻辑设计严密，特别是在**价格红线控制**和**跨模块单据克隆**方面，既保证了经营利润的安全性，又极大提高了业务人员的操作效率。
